### Завдання 2. Реалізація реєстрації та авторизації користувача

Логіка автентифікації та реєстрації реалізована за допомогою патерну MVC (контролери) з використанням бази даних MongoDB та JSON Web Tokens (JWT) для управління сесіями.

#### Реєстрація користувача (Register)
Процес реєстрації обробляється контролером `register`. Основні етапи виконання:
1. **Перевірка унікальності:** Сервер перевіряє базу даних на наявність користувача з таким самим `username`. Якщо такий існує, повертається статус `409 Conflict`.
2. **Хешування пароля:** Для забезпечення безпеки використовується бібліотека `bcryptjs`. Генерується "сіль" (salt) із фактором 10, після чого пароль хешується функцією `bcrypt.hashSync()`. У базу даних зберігається виключно хеш пароля.
3. **Збереження та генерація токена:** Створюється новий документ у колекції `User`. Після успішного збереження генерується `Access Token` за допомогою `jsonwebtoken`, куди в payload "зашивається" ідентифікатор користувача `_id`. Токен дійсний протягом 30 днів.

#### Авторизація користувача (Login)
Процес входу реалізований у контролері `login`:
1. **Пошук користувача:** Здійснюється пошук за введеним `username`. Якщо користувача не знайдено, повертається статус `404 Not Found`.
2. **Верифікація пароля:** Введений користувачем пароль порівнюється зі збереженим у базі хешем за допомогою асинхронного методу `bcrypt.compare()`. У разі неспівпадіння клієнт отримує помилку `400 Bad Request`.
3. **Генерація сесії:** Як і при реєстрації, у разі успішної перевірки генерується JWT-токен на 30 днів, який повертається клієнту разом із даними користувача для подальшої автентифікації запитів.

На стороні клієнта (фронтенд) для управління станом авторизації використовується Redux Toolkit (`authSlice`), а для зворотного зв'язку з користувачем — бібліотека `react-toastify`, яка виводить спливаючі повідомлення про успішний вхід або помилки.

```
//  Register user
export const register = async (req, res) => {
  try {
    const { username, password } = req.body;

    const isUsed = await User.findOne({ username });

    if (isUsed) {
      return res.status(409).json({
        message: 'Даний username вже зайнятий. ',
      });
    }

    const salt = bcrypt.genSaltSync(10);
    const hash = bcrypt.hashSync(password, salt);

    const newUser = new User({
      username,
      password: hash,
    });

    await newUser.save();

    const token = jwt.sign(
      {
        id: newUser._id,
      },
      process.env.JWT_SECRET,
      { expiresIn: '30d' }
    );

    res.status(200).json({
      newUser,
      token,
      message: 'Реєстрація пройшла успішно.',
    });
  } catch (error) {
    res.status(500).json({ message: 'Помилка при створені користувача' });
  }
};

// Login user
export const login = async (req, res) => {
  try {
    const { username, password } = req.body;
    const user = await User.findOne({ username });

    if (!user) {
      return res.status(404).json({
        message: 'Такого корситувача не існує.',
      });
    }

    const isPasswordCorrect = await bcrypt.compare(password, user.password);

    if (!isPasswordCorrect) {
      return res.status(400).json({
        message: 'Неправильний пароль.',
      });
    }

    const token = jwt.sign(
      {
        id: user._id,
      },
      process.env.JWT_SECRET,
      { expiresIn: '30d' }
    );

    res.json({
      token,
      user,
      message: 'Ви увійшли в систему.',
    });
  } catch (error) {
    res.status(500).json({ message: 'Помилка при авторизації' });
  }
};
```

### Завдання 3,7,11. Валідація даних, обробка помилок та збереження користувачів

#### Збереження користувачів у базі даних

Для персистентного зберігання даних користувачів використовується NoSQL база даних MongoDB у зв'язці з ODM-бібліотекою Mongoose. Створено модель та схему `UserSchema`, яка визначає строгу структуру документа користувача у колекції:
* **username**: обов'язкове (`required: true`) та унікальне (`unique: true`) поле рядкового типу.
* **password**: обов'язкове поле рядкового типу, призначене для зберігання криптографічного хешу пароля.
* **posts**: масив посилань (`ObjectId`) на об'єкти моделі `Post`, що встановлює зв'язок між користувачем та його публікаціями.

```
const UserSchema = new mongoose.Schema(
  {
    username: {
      type: String,
      required: true,
      unigue: true,
    },
    password: {
      type: String,
      required: true,
    },
    posts: [
      {
        type: mongoose.Schema.Types.ObjectId,
        ref: 'Post',
      },
    ],
  },
  { timestamps: true }
);

export default mongoose.model('User', UserSchema);
```

#### Клієнтська валідація та підтвердження пароля
Для забезпечення цілісності даних на стороні клієнта (у компоненті реєстрації) впроваджено механізм попередньої перевірки форми перед формуванням HTTP-запиту. 
До інтерфейсу додано додаткове поле для підтвердження пароля. Процес відправки даних блокується з виведенням відповідного попередження, якщо:
* Одне або декілька полів форми (ім'я користувача, пароль, підтвердження пароля) залишаються порожніми.

#### Обробка помилок на сервері

На рівні бекенду (контролери автентифікації) реалізовано надійну обробку виключень з використанням блоків `try...catch`. Це гарантує безперебійну роботу сервера (запобігає крашу процесу Node.js) при виникненні непередбачених ситуацій. Залежно від результату виконання бізнес-логіки, API повертає клієнту відповідні HTTP-коди стану зі зрозумілими повідомленнями у форматі JSON:
* **409 Conflict** — при спробі реєстрації користувача з іменем, яке вже існує в базі даних.
* **404 Not Found** — якщо під час авторизації вказаного користувача не знайдено.
* **400 Bad Request** — у разі неспівпадіння введеного пароля із захешованим значенням у базі.
* **500 Internal Server Error** — при виникненні проблем зі з'єднанням з базою даних або інших внутрішніх помилках сервера.

### Завдання 4,9,12,14,15. Безпека сесій та захист маршрутів та logout

#### Middleware для перевірки токена

Для валідації запитів розроблено проміжне програмне забезпечення (middleware) `checkAuth`. Його головне завдання — перевіряти дійсність JWT-токена перед тим, як дозволити доступ до захищених ресурсів:
* **Витягнення токена:** Функція аналізує HTTP-заголовки запиту (`req.headers.authorization`), вилучаючи токен доступу формату `Bearer`.
* **Верифікація:** За допомогою методу `jwt.verify()` та секретного ключа (`JWT_SECRET`) перевіряється справжність токена та його термін дії.
* **Ідентифікація користувача:** У разі успішної перевірки розшифрований ідентифікатор користувача (`id`) записується в об'єкт запиту (`req.userId`), що дозволяє наступним контролерам знати, хто саме виконує дію. При помилці або відсутності токена запит відхиляється.

```
import jwt from 'jsonwebtoken';

export const checkAuth = (req, res, next) => {
  const token = (req.headers.authorization || '').replace(/Bearer\s?/, '');

  if (token) {
    try {
      const decoded = jwt.verify(token, process.env.JWT_SECRET);

      req.userId = decoded.id;

      next();
    } catch (error) {
      return res.json({
        message: 'Немає доступу.',
      });
    }
  } else {
    return res.json({
      message: 'Немає доступу.',
    });
  }
};

```

#### Реалізація захищених маршрутів

Використання middleware `checkAuth` дозволило створити захищені маршрути (Protected Routes) на рівні бекенду. Наприклад, операції створення, редагування або видалення постів доступні лише після проходження перевірки токена. На рівні клієнта (фронтенду) захист реалізується через умовний рендеринг: посилання на захищені сторінки (наприклад, "Додати пост" або "Мої пости") відображаються у навігаційному меню лише для авторизованих користувачів.

#### Управління сесією та Logout

Логіку завершення сесії (Logout) повністю реалізовано на стороні клієнта у компоненті навігації (`Navbar`). При натисканні на кнопку "Вийти" відбувається наступний ланцюг дій:
1. **Очищення сховища:** Токен доступу видаляється з локального сховища браузера (`window.localStorage.removeItem('token')`), що унеможливлює відправку авторизованих запитів у майбутньому.
2. **Скидання стану:** Викликається Redux-екшен `logout()`, який очищає глобальний стан авторизації у додатку (`authSlice`).
3. **Візуальний відгук:** Інтерфейс миттєво оновлюється (зникають захищені вкладки), а користувач отримує підтвердження про успішний вихід через спливаюче повідомлення.

```
 const logoutHandler = () => {
    dispatch(logout());
    window.localStorage.removeItem('token');
    toast('Ви вийшли з системи');
  };
```

#### Захист від Brute-Force атак (Обмеження спроб входу)

Для підвищення безпеки застосунку та захисту від автоматизованих атак типу перебору паролів (brute-force) було використано бібліотеку `express-rate-limit`. Створено спеціальний middleware `loginLimiter`, який застосовується виключно до маршруту авторизації (`/api/auth/login`). 
Логіка лімітера налаштована таким чином, що з однієї IP-адреси дозволяється виконати не більше 5 спроб входу протягом 15 хвилин. При перевищенні цього ліміту сервер автоматично блокує наступні запити з цієї адреси, повертаючи відповідний статус та попереджувальне повідомлення.

```
const loginLimiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 хвилин
  max: 5,
  message: { message: 'Занадто багато спроб входу. Спробуйте пізніше через 15 хвилин.' }
});
```

#### Архітектура Access та Refresh токенів
Для балансу між безпекою та зручністю користувача стандартна схема JWT-автентифікації була модернізована до системи подвійних токенів (завдання 12):
* **Access Token:** Має короткий термін життя (15 хвилин) і використовується клієнтом для доступу до захищених маршрутів (наприклад, додавання публікацій).
* **Refresh Token:** Має тривалий термін життя (30 днів) і призначений виключно для отримання нової пари токенів після закінчення терміну дії Access Token.

До контролерів автентифікації (`login`, `register`) додано допоміжну функцію `generateTokens`, яка генерує одразу два токени. Також розроблено окремий ендпоінт `/api/auth/refresh`, який приймає діючий Refresh Token, верифікує його валідність (чи не протермінований він та чи існує відповідний користувач) і, у разі успіху, видає клієнту нову, актуальну пару токенів, забезпечуючи безперервну сесію без необхідності повторного введення логіна та пароля.

```
const generateTokens = (userId) => {
  const accessToken = jwt.sign(
    { id: userId },
    process.env.JWT_SECRET,
    { expiresIn: '15m' } // Короткочасний токен (15 хвилин)
  );
  
  const refreshToken = jwt.sign(
    { id: userId },
    process.env.JWT_REFRESH_SECRET || process.env.JWT_SECRET,
    { expiresIn: '30d' } // Довготривалий токен (30 днів)
  );

  return { accessToken, refreshToken };
};
```

### Завдання 10,16,17. Управління профілем користувача

Для надання користувачам повного контролю над їхніми обліковими даними було реалізовано модуль управління профілем. Усі маршрути цього блоку є захищеними і вимагають обов'язкової автентифікації (проходження middleware `checkAuth`).

#### Оновлення даних профілю

Створено ендпоінт (`PUT /api/auth/profile`), який дозволяє користувачу змінювати свої персональні дані, зокрема `username`. На рівні контролера реалізовано додаткову валідацію: перед збереженням нових даних сервер перевіряє базу даних на наявність інших користувачів із таким самим іменем. Якщо ім'я зайняте, повертається помилка `409 Conflict`, що гарантує збереження унікальності записів у БД.

```
// Оновлення профілю (Завдання 10)
export const updateProfile = async (req, res) => {
  try {
    const { username } = req.body;
    const user = await User.findById(req.userId);

    if (!user) {
      return res.status(404).json({ message: 'Користувача не знайдено.' });
    }

    // Якщо юзернейм змінюється, перевіряємо, чи не зайнятий новий
    if (username && username !== user.username) {
      const isUsed = await User.findOne({ username });
      if (isUsed) {
        return res.status(409).json({ message: 'Даний username вже зайнятий.' });
      }
      user.username = username;
    }

    await user.save();

    res.json({ user, message: 'Профіль успішно оновлено.' });
  } catch (error) {
    res.status(500).json({ message: 'Помилка при оновленні профілю.' });
  }
};
```

#### Безпечна зміна пароля

Для реалізації функціоналу зміни пароля (`PUT /api/auth/password`) застосовано підхід із підтвердженням особи. Користувач повинен передати у тілі запиту поточний пароль (`oldPassword`) та новий пароль (`newPassword`). 
1. Сервер знаходить користувача за його ідентифікатором (`req.userId`, отриманим із токена).
2. За допомогою `bcrypt.compare()` перевіряється збіг поточного пароля із захешованим значенням у базі.
3. У разі успіху новий пароль хешується із сіллю (salt) та оновлюється у базі даних.

```
// Зміна пароля (Завдання 16)
export const changePassword = async (req, res) => {
  try {
    const { oldPassword, newPassword } = req.body;
    const user = await User.findById(req.userId);

    if (!user) {
      return res.status(404).json({ message: 'Користувача не знайдено.' });
    }

    // Перевірка старого пароля
    const isPasswordCorrect = await bcrypt.compare(oldPassword, user.password);
    if (!isPasswordCorrect) {
      return res.status(400).json({ message: 'Неправильний поточний пароль.' });
    }

    // Хешування та збереження нового пароля
    const salt = bcrypt.genSaltSync(10);
    user.password = bcrypt.hashSync(newPassword, salt);

    await user.save();

    res.json({ message: 'Пароль успішно змінено.' });
  } catch (error) {
    res.status(500).json({ message: 'Помилка при зміні пароля.' });
  }
};
```

#### Видалення облікового запису
Реалізовано механізм повного видалення профілю (`DELETE /api/auth/account`). Контролер використовує метод `findByIdAndDelete()`, знаходячи користувача за ідентифікатором з його токена. Це унеможливлює видалення чужих акаунтів, оскільки дія прив'язана виключно до поточної авторизованої сесії користувача.

```
// Видалення акаунта (Завдання 17)
export const deleteAccount = async (req, res) => {
  try {
    const user = await User.findByIdAndDelete(req.userId);

    if (!user) {
      return res.status(404).json({ message: 'Користувача не знайдено.' });
    }

    res.json({ message: 'Акаунт успішно видалено.' });
  } catch (error) {
    res.status(500).json({ message: 'Помилка при видаленні акаунта.' });
  }
};
```

У ході виконання даної роботи було успішно спроєктовано та реалізовано захищений вебзастосунок на базі стека MERN (MongoDB, Express, React, Node.js). Практично закріплено навички розробки RESTful API, управління базами даних та побудови архітектури клієнт-серверної взаємодії.

**Основні результати роботи:**

1. **Автентифікація та безпека:** Впроваджено надійну систему реєстрації та авторизації користувачів. Реалізовано механізм сесій на базі JSON Web Tokens (Access та Refresh токени) та застосовано криптографічне хешування паролів за допомогою `bcryptjs`. Додано проміжне програмне забезпечення (`middleware`) для захисту приватних маршрутів та обмеження частоти запитів (`rate-limit`), що забезпечує базовий захист від Brute-Force атак.

Це гарантує збереження лише коректної інформації та ефективно запобігає створенню дублікатів записів (наприклад, реєстрації з уже зайнятим `username`).

3. **Управління станом на клієнті:** На стороні фронтенду успішно інтегровано Redux Toolkit. Використання `createAsyncThunk` дозволило організувати правильну взаємодію з API, безпечно обробляти HTTP-помилки від сервера та забезпечити миттєвий візуальний відгук для користувача.

4. **Забезпечення якості (Quality Assurance):** Проведено комплексне мануальне тестування розробленого API за допомогою інструментарію Postman.
